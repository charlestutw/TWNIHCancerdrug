# nhi-ch9-data.json 維護說明小抄

給未來任何維護者（人或 AI session）：本檔說明如何安全更新健保第九節抗癌藥物速查表的資料。
**只要改這一份 JSON，所有裝置下次開啟就自動更新，HTML 不必重發。**

---

## 1. 檔案結構總覽

```json
{
  "version": "2025-06b",
  "updated": "含 115 年 6 月修訂版",
  "source": "衛生福利部中央健康保險署 · 藥品給付規定第九節",
  "groups": { "肺癌": ["肺癌"], "...": ["..."] },
  "drugs": [ { ...藥品物件... }, ... ]
}
```

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `version` | string | ✅ | **更新時務必往後推**。格式 `YYYY-MM`，程式用**字串比大小**判斷新舊。 |
| `updated` | string | ✅ | 顯示在標題列的人類可讀說明。 |
| `source` | string | ✅ | 資料來源標註。 |
| `groups` | object | ✅ | 癌別分群定義（分群鈕名稱 → 癌別標籤陣列）。詳見第 3 節。 |
| `drugs` | array | ✅ | 藥品陣列，目前 122 筆。 |

### ⚠️ version 是自動更新的唯一開關

程式邏輯是 `remoteVersion > localVersion`（字串比較）才更新。

- ✅ `"2025-06"` → `"2025-09"`　會更新
- ✅ `"2025-06"` → `"2026-01"`　會更新
- ❌ `"2025-06"` → `"2025-6"`　**不會更新**（少了前導 0，`"6" > "0"` 但長度錯位，且破壞格式一致性）
- ❌ 忘記改 version　**不會更新**（使用者手動按「檢查更新」才會強制拉取）

**同月內二修**：用 `"2025-06b"`、`"2025-06c"`（`b > ""`，字串比較仍成立）。

## 2. 藥品物件欄位定義

每個 `drugs[]` 元素恰好 7 個欄位，**不多不少**：

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `id` | string | ✅ | 條文號碼，如 `"9.80"`、`"9.32.1"`。**必須唯一**。 |
| `name` | string | ✅ | 藥品學名，如 `"Osimertinib"`。全文/精確搜尋都會比對。 |
| `brand` | string | ✅ | 商品名，如 `"Tagrisso"`。**無商品名請填空字串 `""`，不要省略、不要填 null**（目前有 7 筆為空）。 |
| `type` | string | ✅ | 藥品類型，**只能是下列五個字串之一**（見下表）。 |
| `cancers` | string[] | ✅ | 癌別標籤陣列，至少 1 個。**用詞必須與既有標籤一致**（見第 3 節）。 |
| `ind` | string | ✅ | 給付適應症。`verbatim:true` 時為**官方條文原文，逐字保留，嚴禁改寫**。 |
| `req` | string | ✅ | `verbatim:true` 時為出處說明；否則為申請條件摘要。 |
| `verbatim` | boolean | ✅ | `true` = `ind` 為官方 PDF 原文；`false` = 人工摘要。 |
| `eff` | string | ✅ | 原文生效日 `YYYY-MM-DD`。非 verbatim 填 `""`。 |
| `src` | string | ✅ | 來源 PDF 檔名。非 verbatim 填 `""`。 |

### `type` 合法值（區分大小寫，全大寫）

| 值 | 顯示 | 顏色 | 目前筆數 |
|---|---|---|---|
| `TARGETED` | 標靶 | 綠 | 74 |
| `CHEMO` | 化療 | 藍 | 31 |
| `HORMONE` | 荷爾蒙 | 金 | 11 |
| `IMMUNO` | 免疫 | 紫 | 5 |
| `CART` | CAR-T | 紅 | 1 |

填入其他值（如 `"targeted"`、`"標靶"`）→ **卡片渲染會直接壞掉（讀不到顏色）**。

## 3. `cancers` 標籤與癌別分群（最容易踩的坑）

### 單一事實來源

分群定義**就寫在這份 JSON 的 `groups` 欄位裡**。HTML 與 `validate-data.js` 都從這裡讀取，所以**不可能再互相漂移**。

```json
"groups": {
  "肺癌": ["肺癌"],
  "消化道": ["大腸直腸癌", "胃癌", "食道癌", "胰臟癌", "膽道癌", "GIST", "胰臟神經內分泌腫瘤"]
}
```

**這代表：改分群現在只要改 JSON，不必重發 HTML。** 新分群會隨自動更新推送到所有裝置。

規則：
- 鍵 = 畫面上的分群鈕名稱；值 = 該鈕收錄的 `cancers` 標籤陣列
- 不可使用保留名稱 `"全部"`（介面自動加入）
- 同一標籤可出現在多個分群（會被多個鈕篩到），驗證器會發**警告**提醒
- 藥品的 `cancers` 字串必須**精確等於** `groups` 中的某個標籤，否則按分群鈕篩不到（全文搜尋仍找得到）

> HTML 內另有 `FALLBACK_GROUPS`，**僅在遠端 JSON 缺少 `groups` 欄位時**啟用，確保舊版資料不會讓介面壞掉。正常情況下永遠不會用到，也不需要維護它。

### ✅ 已修正的缺陷（歷史紀錄，勿重蹈）

初版有兩類錯誤，已於 `2025-06b` 修正：

- **孤兒標籤**（存在於 JSON、卻沒被任何分群收錄）：
  `頭頸癌`（新增專屬分群鈕）、`膀胱癌`（併入 婦科/泌尿）、`卡波西肉瘤`（併入 皮膚/其他）、`胰臟神經內分泌腫瘤`（併入 消化道）
- **幽靈標籤**（分群列了但資料裡沒有）：`泌尿道腫瘤` —— 已刪除，實際資料一律用 `泌尿道上皮癌`
- **結構性根因**：`GROUPS` 原本在 HTML 與驗證器各抄一份，會悄悄漂移。**已把 `groups` 收進 JSON 作為單一事實來源。**

> **教訓**：任何「同一份設定抄兩份」的結構都會漂移。發現時要收斂成單一來源，而不是靠人工同步。

### 新增癌別的規則

1. 優先**沿用既有 34 個標籤**（見上表）。
2. 真的需要新標籤 → 在 JSON 的 `groups` 中把它加進某一群（**只需改這一處**）。
3. 跑 `node validate-data.js nhi-ch9-data.json` 確認零警告。

## 4. `ind` 欄位的排版語法

`ind` 是純字串，但渲染時有兩個轉換規則：

| 你寫的 | 畫面呈現 |
|---|---|
| 全形分號 `；` | 自動換行成新的一條 `• ` 項目 |
| 換行符 `\n` | 保留為換行（JSON 內寫 `\n`） |
| `★` | 自動轉成**金色粗體 ★**，用來標示擇一使用規則 |

**慣例**：主要適應症用 `；` 分隔；擇一使用規則另起一行，以 `★` 開頭。

```
"ind": "EGFR Exon 19 Del 或 Exon 21 L858R突變之肺腺癌第一線；先前EGFR TKI失敗且有T790M突變之NSCLC第二線\n★與gefitinib/erlotinib/afatinib/dacomitinib擇一（第一線時）"
```

目前 53 筆含換行、48 筆含 `★`。

> ⚠️ 半形分號 `;` **不會**觸發換行。中文條文一律用全形 `；`。

## 5. 新增一筆藥品：完整範例

假設健保署新增 **9.119 Amivantamab（Rybrevant）**，用於 EGFR exon 20 插入突變 NSCLC。

**步驟：**
1. 打開 `nhi-ch9-data.json`
2. 在 `drugs` 陣列**末尾**（`9.118` 之後）插入下列物件，注意前一筆結尾要補逗號
3. 把 `version` 從 `"2025-06"` 改成新版本（如 `"2025-09"`）
4. 同步更新 `updated` 文字
5. 存檔 → 上傳覆蓋雲端那份 → 完成

```json
{
  "id": "9.119",
  "name": "Amivantamab",
  "brand": "Rybrevant",
  "type": "TARGETED",
  "cancers": ["肺癌"],
  "ind": "+carboplatin+pemetrexed用於EGFR exon 20插入突變之局部晚期或轉移性NSCLC成人第一線；含鉑化療後疾病惡化之EGFR exon 20插入突變NSCLC\n★與其他EGFR-TKI擇一使用",
  "req": "需EGFR exon 20 插入突變檢測報告（通則十二）；需事前審查（每3個月申請一次）"
}
```

同時把檔案頂端改成：

```json
"version": "2025-09",
"updated": "含 115 年 9 月修訂版",
```

### 其他常見操作

**修改現有藥品**：直接改該物件的 `ind` / `req`，記得推 `version`。

**刪除藥品（健保取消給付）**：整個物件移除，注意逗號。務必推 `version`。

**新增子條文**（如 `9.119.1` / `9.119.2`）：`id` 直接用三段式即可，程式無特殊處理，只要唯一。

## 6. 存檔前檢查清單

- [ ] JSON 語法合法（最常見錯誤：**多餘或缺少的逗號**、中文引號 `"` 誤用成 `“”`）
- [ ] `version` 已往後推，且格式為 `YYYY-MM`
- [ ] 新增的 `id` 沒有跟既有重複
- [ ] `type` 是五個合法值之一（全大寫）
- [ ] `brand` 若無商品名，是 `""` 而非 `null` 或缺欄
- [ ] `cancers` 每個字串都能在 `groups` 中找到（否則按分群鈕篩不到）
- [ ] 若改了 `groups`：沒有使用保留名稱 `"全部"`，且每群至少 1 個標籤
- [ ] `ind` 內用全形 `；`，擇一規則用 `★` 開頭並置於 `\n` 之後
- [ ] 七個欄位齊全，沒有多餘欄位

### 自動驗證（有 Node.js 環境時）

把同目錄下的 `validate-data.js` 跑一次，會一次檢出上述所有問題：

```bash
node validate-data.js nhi-ch9-data.json
```

沒有 Node 環境時，最低限度請把 JSON 貼進任一 JSON 驗證器確認語法合法。

## 8. 給 AI session 的注意事項

- 本 JSON 是**臨床參考資料**。任何條文內容的異動，來源必須是**健保署正式公告**，不得依模型記憶推測。查不到就標註「待查證」，不要編造給付條件、療程上限或基因檢測要求。
- 健保署只發布 PDF（有機器人防護、網址會變動），**沒有可自動爬取的 API**。內容更新必然需要人工／AI 閱讀公告後手動落檔，這是設計上的已知極限，不是缺陷。
- 增修 `drugs` 內容可直接進行。修改 `groups`（會改變畫面上的篩選鈕）或 HTML 的 `TYPES`／介面邏輯屬於**改動程式行為**，動之前請先向使用者確認。
- 每次踩到新的坑（例如又發現孤兒標籤），請把教訓寫回本檔第 3 節的「已知缺陷」，不要只在對話裡講。

## 9. 每月更新流程（半自動）

### 能自動的：偵測「哪一條被改了」

健保署有開放資料 REST API（免金鑰、每月更新）。每個給付規定條文對應一份官方 PDF，**檔名內含生效日期**，例如 `9.22._20240601.pdf`。生效日一變，就代表該條文被修訂。

- `monitor-nhi.js` — 抓品項表 → 取出第九節條文與生效日 → 與 `clause-index.json` 快照比對 → 報出**修訂／新增／消失**
- `monthly-check.yml` — GitHub Actions，每月 5 日執行；有異動就自動開 issue、附報告，並把官方 PDF commit 進 `clause-text/`

```bash
node monitor-nhi.js --init        # 首次建立快照
node monitor-nhi.js               # 比對（有異動時退出碼 1）
node monitor-nhi.js --fetch-text  # 順便下載異動條文 PDF
```

### ⚠️ 覆蓋率限制：只有 10 / 118 條

開放資料的品項表**只涵蓋部分口服藥**。實測 `Osimertinib`、`Trastuzumab`、`Pembrolizumab`、`Olaparib`、`Docetaxel`、`Bevacizumab` 等**皆不在資料集內**。

目前可自動監測的條文僅：
`9.11. 9.13. 9.22. 9.30. 9.32. 9.32.1. 9.32.2. 9.33. 9.50. 9.52.`

**其餘 108 條（含絕大多數標靶、免疫、注射劑）仍須人工核對健保署公告。** 這是資料源的極限，不是腳本缺陷。

### 不能自動的：內容改寫

把官方條文濃縮成 `ind` / `req` 的速查用語，涉及取捨與壓縮判斷（哪些條件必列、`★` 擇一規則怎麼標），**必須由人或 AI 讀過官方 PDF 後改寫**。任何宣稱能全自動產生臨床參考內容的做法都不可信。

### 實際流程

```
GitHub Actions 每月執行
   ↓ 偵測到 9.xx 生效日變動
自動開 issue，附官方 PDF
   ↓
人或 AI session 讀 PDF → 更新 nhi-ch9-data.json 的 ind / req
   ↓
推進 version → node validate-data.js（必須零錯誤）
   ↓
mv clause-index.json.new clause-index.json → commit
   ↓
上傳覆蓋 Gist → 所有裝置自動同步
```

## 10. 官方原文（verbatim）機制

### 現況：90 / 122 條為官方原文

`verbatim: true` 的條目，`ind` 直接是健保署條文 PDF 抽出的**逐字原文**，介面上以綠色徽章「官方原文 · 生效日 YYYY-MM-DD」標示，並用等寬區塊呈現。

`verbatim: false` 的 32 條，健保署**未發布獨立條文 PDF**（已實測 2002–2026 全區間皆無），只能沿用人工摘要，介面上以黃色徽章「摘要（非原文）」明確標示，不會冒充原文。

未提供 PDF 的 32 條：
`9.1.1 9.1.2 9.1.3 9.6 9.8 9.9 9.11 9.12.1 9.13 9.14 9.15 9.16 9.17 9.19 9.21 9.23 9.25 9.33 9.35 9.36.1 9.36.2 9.38 9.39 9.40 9.48 9.57 9.58 9.65 9.68 9.86 9.88 9.92`

### 🚫 verbatim 條文的紅線

- **`ind` 一字都不可改**。不可濃縮、不可補字、不可換標點、不可加 `★`。
  介面對 verbatim 條文**不套用** `；` → 條列 的排版轉換，正是為了避免竄改原文。
- 要更新原文，唯一正當做法是**重抓官方 PDF**：
  ```bash
  node harvest-official.js     # 可中斷續跑，已抓過的會跳過
  ```
  刪掉 `official-text.json` 中某條，即可強制重抓該條。
- `src` 與 `eff` 必須與實際抓到的檔案一致，供日後查證。

### 取得原文的原理

條文 PDF 端點：
```
https://info.nhi.gov.tw/api/INAE3000/INAE3000S01/getPDF?DurgFileName=<條文代碼>._<YYYYMMDD>.pdf
```
生效日一律為某月 1 日，且每條文只有一份現行 PDF。`harvest-official.js` 由新到舊掃描月份，命中即停；子條文找不到時退回上層條文（如 `9.5.2` → `9.5.`）。

> **教訓**：能拿到官方原文時，絕不要用模型改寫過的版本。臨床參考資料的價值來自可查證性，不是可讀性。摘要與原文必須在介面上一眼可辨。
